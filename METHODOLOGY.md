# Frontend Requirement Mapper — 方法论

> 本文记录了这个 Skill 的设计思路：为什么要建它、工作流背后的判断逻辑，以及它能产出什么。

---

## 一、痛点：追齐需求为什么这么累

追齐类需求有一个共同特征：**PRD 是残缺的**。产品同学写的是意图，不是实现——"对齐追齐APP的商品详情页"，具体对齐哪些字段、哪些坑位、哪些状态，PRD 里没有。

于是开发者不得不自己补全信息，实际工作量远超 PRD 描述的范围：

**1. 要自己扒参考端代码**

PRD 说"参考追齐APP"，但追齐APP的代码不是你维护的。需要自己找到对应页面的入口、组件、数据来源、字段定义，还要搞清楚展示条件、状态变体、埋点逻辑——这些都需要逐文件翻。

**2. 要自己扒目标端代码**

知道参考端做了什么之后，还要摸清自己负责的仓库里对应模块在哪、现在是什么状态、哪些能复用、哪些要改、哪些要新写。有时候同一个功能点散落在多个文件里，找齐就要花不少时间。

**3. 两边都扒完还要对比**

参考端的字段叫 `imageUrl`，目标端叫 `mainPic`，是同一个东西吗？参考端从服务 A 拿数据，目标端从服务 B 拿，字段结构不一样怎么办？这些对比是脑力活，容易出错，也容易遗漏。

**4. 有时候不止一个仓库**

追齐需求经常涉及多端：主站 H5、小程序、BFF、有时还有活动页。每个仓库都要找对应位置、看现状、做对比——乘以 N 个仓库，工作量线性增长。

**5. 做完分析才发现漏了**

等代码写完了、进了测试，才发现某个状态没处理、某个字段服务端根本不返回、某个坑位在特定条件下展示逻辑不对。这些返工成本很高，但在需求分析阶段本来是可以提前发现的。

---

**这个 Skill 的出发点是**：把上述手工扒代码、手工对比、手工整理的过程，交给 Agent 来完成。开发者只需要提供 PRD、仓库路径，以及自己已知的关键信息（比如代码在哪几个文件），Agent 系统地跑完分析流程，输出一份可以直接用于评审和实现的需求分析报告。

---

## 二、工作流拆解与思考点

### 总体设计原则

**证据先于结论**：每个结论都要有文件路径、字段名、组件名作为支撑。没有证据的结论标为 `推测` 或 `待确认`，不进报告。

**diff 驱动 > PRD 驱动**：先把两个 repo 之间的结构差异全貌掌握，再按风险顺序深挖——而不是只找 PRD 提到的点。

**人工知识 > Agent 搜索**：开发者一行提示（"目标端代码在 `src/pages/goods/detail/`"）比 Agent 做几十次关键词搜索更快、更准。人工介入的正确时机是在最开始，而不是在 Agent 卡住之后。

**分层处理，按需展开**：不强迫所有任务走完整流程。小规模逻辑追齐只需要前几个步骤；大规模跨栈 UI 改动才需要全量走。

---

### Stage 0–2：理解需求，建立检索基础

**思考点**：直接在代码里搜 PRD 原文是反模式。PRD 用业务语言，代码用技术语言，两者之间存在命名断层。正确的顺序是：

1. 先建立业务词典（`金刚区` = `kingkong / grid / shortcut`）
2. 把模糊的 PRD 转化成开发可执行的 Spec（标出 `已明确 / 待确认 / 缺失定义`）
3. 用 Spec 里的字段、页面名、坑位名构建检索词表
4. 用检索词表去找代码，而不是用原始 PRD

这三个阶段是"翻译"过程，不是可以跳过的前置工作。

---

### Stage 3：参考 APP 逆向

**思考点**：逆向的目标不是"参考端有什么功能"，而是"参考端实际做了什么"——PRD 描述和实现之间可能有差距，以代码为准。

逆向要追到底：路由 → 页面容器 → 子组件 → 数据来源 → 字段 → 条件 → 埋点。只看 UI 文件就停下来，是最常见的浅尝辄止错误。

---

### Stage 4：跨仓库结构 Diff（核心新增阶段）

这是整个 Skill 设计中最重要的一个判断：**在进入目标项目映射之前，必须先把两个 repo 之间的差异地图画出来。**

#### Pre-flight：Diff 前的准备

直接 diff 两个陌生的 repo 会产生噪音，因为目录结构的语义在不同技术栈下完全不同。Diff 之前需要三件事：

1. **技术栈快照**：两端各用什么框架、路由、状态管理、API 层、BFF——建立"同一个概念在两端各长什么样"的翻译键

2. **目录约定对齐**：`pages/` 对应 `screens/`，`api/` 对应 `services/`——这张表是后续所有路径比较的基础

3. **模块锚点声明**：声明每个功能模块在两个 repo 里各在哪。**首选是开发者直接提供**，比 Agent 搜索更快更准。提供一行 `参考端：src/screens/Product/Detail/，目标端：src/pages/goods/detail/` 就可以跳过整个模块发现阶段。

#### 服务对齐分类

在做字段 diff 之前，先判断两端的服务关系：

| Case | 服务关系 | 字段情况 | 意味着 |
|------|---------|---------|-------|
| Case 1 | 同服务、同 BFF | 字段一致 | 只需 diff 前端结构 |
| Case 2 | 不同 BFF，上游服务相同 | 字段基本一致 | 前端 + BFF 轻改；查 BFF 代码推断字段能否透传 |
| Case 3 | 不同 BFF，上游服务不同 | 字段差异大 | 前端 + BFF + 完整字段重映射；`字段缺失` 需上升后端 |

这个分类决定了 diff 的深度和 BFF 是否在改动范围内。前端需求分析不只是客户端——BFF 改动也是需求的一部分。

#### Response 报文直接 diff

当开发者能提供两端同一页面的实际 API 返回 JSON 时，优先用报文做字段 diff，而不是从 TypeScript 接口或代码推断。

理由：接口定义可能过时，BFF 的字段变换很难从静态代码追踪，实际报文反映的才是 runtime 真相。

报文 diff 还能发现一类隐蔽问题：**类型不一致**。`price: 1200`（分）和 `price: "¥12.00"` 字段名相同、语义相近，但直接替换会出 bug。

#### 跨栈 UI Diff：比语义契约，不比代码

React JSX 和 WXML 语法不兼容，字段名可能不同（两端调用不同服务），className 不同——代码 diff 是纯噪音。

正确的做法是提取双方的 **UI contract（语义契约）**，diff 契约而不是代码：

- 这个坑位存在吗？
- 由哪个字段驱动？
- 可见性条件是什么？
- 挂了什么交互？

元素寻址用**视觉语义路径**（`商品详情页 > 商品卡片 > 右上角促销角标`），而不是 DOM 路径或字段名——这是唯一跨栈稳定的锚点。两端的字段各自独立发现，不做跨端字段名假设。

当 app 可以运行时，优先用**运行时辅助 diff**：截图对比找出"什么不一样"，浏览器 DevTools（React/Vue DevTools + Playwright）定位"在哪"——把过去靠猜测的 UI 定位变成有工具辅助的流程。小程序和 RN 因平台限制，退化为静态代码阅读。

---

### Stage 5：目标项目映射

**思考点**：映射的起点是 Diff Map，而不是 PRD。按风险排序处理每一行，而不是按 PRD 提及顺序。

`直接复用` 的行快速确认后跳过；`字段缺失` 的行立即上升，不等到最后。

---

### Stage 8：QA 静态验证

验证分三层，边界非常清晰：

| 层 | 谁做 | 能做什么 |
|----|-----|---------|
| **逻辑验证** | Agent 静态 | 校验函数、正则、类型系统、maxlength |
| **UI 结构验证** | Agent 静态 | 坑位存在、字段绑定、条件渲染、事件挂载、状态变体 |
| **视觉验证** | 人工 + 截图工具 | 布局、间距、颜色——只有跑起来才能看 |

Agent 做完结构验证后，输出一个**精准的视觉确认清单**：哪个坑位、为什么需要人工看。不给"所有 UI 需要视觉验证"这种无意义的笼统结论。

## 三、产物

每次运行 Skill 的完整产出包括：

### 过程产物（中间态，辅助判断）

| 产物 | 用途 |
|------|------|
| 业务词典 | 对齐 PRD、代码、团队口径 |
| Spec 初稿 | PRD 转化为开发可执行的定义 |
| 模块锚点表 | 两端功能模块位置对照 |
| Diff Map 表 | 全量差异地图（变更类型 + 初步分类） |
| UI contract 对比表 | 跨栈场景下的坑位/字段/条件语义对比 |
| 字段名映射表 | 两端字段对应关系（含改名、结构差异、缺失） |

### 最终产物

**需求分析报告**（基于 `assets/requirement-analysis-template.md`）：

- 需求摘要与背景
- PRD 澄清结果（已明确 / 待确认 / 潜在风险）
- 参考 APP 功能盘点
- 目标项目现状与差异分析
- 字段映射
- 改造范围（页面、组件、hooks/store、API、埋点、BFF）
- 风险与易漏项
- 待确认问题列表
- 开发任务拆解
- 测试点

**QA 验证记录**（基于 `assets/qa-record-template.md`）：

- 逻辑层验证结果（逐 case）
- UI 结构验证结果（逐坑位）
- 视觉验证待确认清单（精准到坑位 + 原因）

## 四、这套方法论的核心判断

回顾整个设计过程，有几条反复出现的判断在驱动每一次迭代：

**语义锚点 > 语法锚点**
跨栈时，字段名、DOM 路径、class 名都是不稳定的。唯一稳定的是视觉语义：用户看到什么、在哪、什么时候。

**人工提供 > Agent 搜索**
开发者知道代码在哪，Agent 不知道。让 Agent 靠关键词猜模块位置，不如让开发者在 PRD 调研阶段顺手说一句。这是成本最低、收益最高的人机协作点。

**先问"能不能跑"**
能跑的 app 用截图 + 浏览器 DevTools；不能跑的才读代码。运行时信息（实际 response、截图）比静态代码更接近真相。

**分层，明确边界**
Agent 做得好的（逻辑验证、结构验证），和人做得好的（视觉确认、业务判断），要明确分开。不要让 Agent 做超出能力边界的事，也不要让人做 Agent 能做的事。

**复杂度按需展开**
小任务不走大流程。决策树和文件拆分的本质是一样的：让简单场景保持简单，让复杂场景有路可走。
